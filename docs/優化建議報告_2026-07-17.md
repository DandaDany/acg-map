## 一、使用者體驗（UX）

### 1.1 「為什麼別人會用？」——先承認殘酷的現實

目前這題**答不出來**，原因有三層：

1. **可能還沒真的能用**：`DECISIONS.md` 自己都標註「⚠️ 目前未確認已實際部署，僅為建議」。一個沒有公開網址、要雙擊 HTML 或本機開 `http.server` 才能看的東西，對「別人」而言不存在。**這是第一優先要解掉的**——先有一個穩定公開網址，其他優化才有意義。
2. **內容密度撐不起「地圖」的期待**：57 個場館、151 場活動、集中在雙北高中（台北 23、高雄 10、台中 9）。使用者打開地圖，期待的是「全台密密麻麻的 ACG 活動」，看到的是稀疏幾個點。第一印象就會流失。
3. **沒有「只有這裡才有」的理由**：ACG 玩家現在去哪找展覽/快閃？IG hashtag、巴哈姆特、Facebook 社團、KKTIX/Accupass。本站要贏，唯一的護城河是**「一張地圖看完全台、且資訊比社群貼文更結構化（時間/地點/導航/收藏）」**。這個定位是對的，但目前執行沒有把它做滿（見下）。

> **建議**：在 README 與網站首屏用一句話講清楚價值主張，例如：「**全台 ACG 展覽・快閃・主題餐廳，一張地圖看完，想去的直接收藏、一鍵導航。**」目前站名有講「ACG 活動地圖」，但沒有把「為什麼比滑 IG 好」講出來。

### 1.2 哪些人會用？他們需要什麼？——TA 沒有被定義

專案文件通篇在講資料管線，但沒有一份「使用者輪廓」。我幫你補上，這會直接影響功能優先序：

| TA | 情境 | 最在意 | 目前滿足度 |
|----|------|--------|-----------|
| **本地追展族**（主力） | 週末想找附近有什麼展/快閃 | 「附近」「這週還在的」「值不值得跑一趟」 | 中：有時間篩選、有附近 2km，但缺活動詳情/評價/圖 |
| **外地/觀光客** | 來某城市玩，順路看 ACG 活動 | 依城市快速掃、營業時間、交通 | 低：有地區篩選，但無營業時間、無交通資訊 |
| **限定商品狩獵者** | 快閃店/主題餐廳限定商品 | 開始/結束日、是否需預約、有沒有限量 | 低：有起訖日但無「需預約」「限量」等關鍵欄位 |
| **回訪規劃者** | 之前看到想去、之後回來查 | **收藏能留住** | **失敗**：收藏會蒸發（見 1.4） |

**共通的三個核心需求，目前都沒做滿**：
- **「留住」**：收藏壞掉（1.4）。
- **「提醒」**：完全沒有。快閃店會結束，使用者需要「這活動剩 3 天」「加入行事曆」。目前 30 場活動連結束日都沒有，`即將結束` 篩選在現資料下永遠是空的（交接文件自己也記了這點）。
- **「決定要不要去」**：活動卡沒有描述文字（`ex[].h` 欄位 **151 場全部為空**），只有標題＋日期＋一張（可能過期的）圖。資訊量不足以支撐決策。

### 1.3 缺少的關鍵功能（依投報比排序）

**P0（不做等於沒產品）**
1. **正式部署 + 穩定網址**（見第三章）。
2. **修好收藏**（見 1.4）——這是使用者親口說的痛點。

**P1（決定留存率）**
3. **輕量帳號 / 跨裝置同步**：不必自建後端。最省力方案是「**匯出/匯入收藏碼**」（把 localStorage 收藏序列化成一段可複製字串或網址參數），先解決「換裝置/清快取就沒了」。想做真同步，可接免後端的 BaaS（Supabase / Firebase Auth 免費額度），或用「Google 一鍵登入 + 存一個 JSON 到雲端」。**但這在 1.4 修好之前不要做**，否則同步的是一堆會失效的鍵。
4. **活動詳情**：填 `ex[].h`（一句話簡介）、加「是否需預約」「票價/免費」「營業時間」欄位。這需要 schema 擴充（交接文件第三階段待辦，且 `DECISIONS.md` 說 schema 變更要先確認，屬架構級）。
5. **提醒 / 加入行事曆**：活動卡加「加入 Google 日曆」連結（純前端 `.ics` 或 calendar URL 即可，零後端）。這對「快閃店會結束」的場景是剛需。

**P2（加分）**
6. **列表/清單視圖**：地圖適合探索，但「這週全部活動」用列表更快。目前只有地圖。
7. **分享單一活動**：可用 URL hash（`#event=<id>`）深連結到某活動並自動開 popup。目前分享只能貼整站首頁。
8. **手機體驗**：見第三章 a11y。

### 1.4 收藏「雞肋」的根因與解法（重點）

使用者說「有收藏，但沒辦法登入，更新網頁收藏又不見了」。這其實是**兩個獨立問題**，且第二個是可修的 bug：

**問題 A：無帳號、不跨裝置**（已知取捨，見程式註解 `免登入、不跨裝置、不做社群/分享`）。這是設計選擇，用 1.3-3 的匯出碼/輕量登入解。

**問題 B：收藏會「蒸發」——真正的 bug**。程式在 `taiwan-exhibition-map.html`：
```js
function favKey(e,v){ return [(v&&v.name)||'', e&&e.t||'', e&&e.s||'', e&&e.l||''].join('¦'); }
```
收藏鍵 = `場館名 ¦ 活動標題 ¦ 開始日 ¦ 活動連結`。這是**內容衍生鍵**。每次 `backend/update_all.py` 重建 `venues.json`：
- 活動標題被重新抓取/正規化（多一個空白、括號全半形變動）→ `e.t` 變 → 鍵變。
- 連結被 `resolve_short_urls.py` 展開、或 IG 連結帶新參數 → `e.l` 變 → 鍵變。
- 開始日格式或補值 → `e.s` 變 → 鍵變。

只要任一欄變動，localStorage 裡存的舊鍵就**對不上任何活動**，`collectFavItems()` 撈不到，收藏就消失了。使用者感受到的「更新網頁收藏又不見了」，很可能不是 localStorage 被清，而是**資料一更新，鍵就失配**。

**解法（依序）**：
1. **給每場活動一個穩定 ID**：在 `refresh_venues.py` 產出時，為每個 event 算一個**不隨排版變動的穩定鍵**（例如 `場館名 + 正規化後標題 + 開始日` 做 hash，且標題正規化要 trim/統一標點/去掉 tracking 參數），寫進 `ex[].id`。前端 `favKey` 改成直接用 `e.id`。這樣資料更新只要活動還在，ID 就不變，收藏就不掉。
2. **收藏遷移相容**：發佈新版時，讀到舊格式鍵先嘗試 best-effort 對映到新 ID，避免既有使用者一次全掉。
3. **匯出/匯入**：加「複製我的收藏」→ 產生短碼或 `?favs=` 網址，讓使用者能備份/換機。

> 這題修好，「雞肋」感會消失一大半，且是純前端 + 一個後台欄位，成本低、投報比最高。

## 二、自動化邏輯

### 2.1 現況流程還原（依程式與文件）

- **主資料管線**：`ops/run_scheduled_update.sh` → `backend/update_all.py`，共 13 步（collect_venues → Excel 清洗/匯入 → CACO/Cayenne → logo → refresh 第一輪 → FB logo → geocode → KV → refresh 第二輪 → 縮圖 → 同步 HTML 內嵌 DATA）。
- **排程載體**：macOS **launchd**，每週一、四 11:30，跑在**你的個人 Mac**（`/Users/daniel0522/...`）。有 lock、有 6 小時 skip window、有 log，這部分做得不錯。
- **維護回報**：`report_status.py` 每週一/四由 Claude 排程跑，比對上次快照，列出「缺 KV / 約略定位 / 缺連結 / 疑似漏判 ACG」的缺口。
- **人在迴圈**：排程抓完 → 你請 Claude cowork 依回報補缺口（補 KV、補分類 override、補 Excel）→ 再跑一次。每天另外請 cowork 去爬新案例。

### 2.2 更新頻率夠嗎？

**分內容型態看，一刀切的「每週兩次」不夠精準**：
- **常設展/長期展**：每週兩次綽綽有餘，甚至太頻繁。
- **快閃店 / 主題餐廳**：這類「限定、會突然開、突然收」，每週兩次**偏慢**。使用者最怕「跑去發現已經結束/還沒開」。這類理想是每日。
- **新活動上架**：靠「每天請 cowork 爬案例」補，但這是手動觸發、非排程，**穩定性取決於你有沒有記得做**。

**結論**：頻率本身（週二次）對展覽 OK，但對 ACG 主打的快閃/餐廳偏慢，且「每日爬案例」沒有自動化、靠人記得。

### 2.3 流程順嗎？——脆弱點盤點

| 脆弱點 | 說明 | 風險 |
|--------|------|------|
| **綁死個人 Mac** | launchd 只在你 Mac 開機且到點時跑。闔上蓋、關機、出門、Mac 壞了 → 完全停更。 | 🔴 單點故障，且對「線上使用者」不可見地變舊 |
| **雲端跑不到 `.gov.tw`** | 文件明載沙箱連不到多數政府站，所以才留在 Mac。等於**無法搬上 CI**（用現有來源的話）。 | 🔴 架構被綁住 |
| **失敗即停、無告警** | `update_all.py` 任一步非 0 就 `sys.exit`，`venues.json` 維持上一版。好處是不產半套資料；壞處是**你不會被主動通知失敗**，只能事後翻 log。 | 🟠 靜默停更 |
| **KV 用 IG 圖床** | `img` 指向 `scontent.cdninstagram.com`（帶 `oe=` 到期）。過期後前端破圖，且下次管線的圖片驗證可能把它清成空。 | 🟠 圖片會慢慢爛掉 |
| **分類靠關鍵字表** | `classify_theme_kw` 用關鍵字判斷是否 ACG；新 IP/角色不在表裡就被歸「其他文化」而漏掉。所以才需要每次人工「分類複核」。 | 🟠 漏收 ACG，且靠人補 |
| **cowork 補件無護欄** | 人在迴圈補 KV/分類/Excel 很聰明，但**沒有規則約束**（日誌就出現過 B-SIDE LABEL 一下排除、一下又補進來的矛盾）。 | 🟡 資料判斷不一致、難稽核 |

### 2.4 建議：把「機械活」自動化、把「cowork」升級為審核者

**方向：不是叫 cowork 更勤勞，而是把確定性的工作變成程式、把 cowork 留給判斷。**

1. **把可搬的來源搬上 GitHub Actions（CI）**：
   - 文化部 API、CACO、Cayenne、Accupass、多數官網其實在雲端可連（真正連不到的只有部分 `.gov.tw`）。先盤點「哪些來源沙箱連得到」，把這部分改成 **GitHub Actions 排程（cron）每日跑**，產出直接 commit `venues.json` 並自動部署（見第三章）。
   - 真正只能在 Mac 跑的少數 `.gov.tw` 來源，留在 Mac，但把 Mac 的角色縮到最小。
   - 好處：Mac 關機也會更新；有 CI log、有失敗通知（Actions 失敗會寄信/發通知）。
2. **加「資料異常偵測」當守門員**（交接文件第三階段本來就列了，優先做）：
   - 場館/活動數較上一版**驟降 > X%** → 擋下並告警（避免像政府 API 關掉那次，靜默從 199 掉到 44 才發現）。
   - 日期異常（結束日早於開始日、開始日在過去很久）、重複活動、必要欄位（起訖日）缺漏 → 標記。
   - 這支腳本 = 每次 refresh 後跑，異常就**寫進回報並可選擇擋 commit**。
3. **重新定位 cowork 的角色**：從「每天去爬」改成「**每天審一份自動產生的候選清單**」。
   - 自動：CI 爬到新活動 → 丟進 `data/manual/候選待審.json`（帶來源、信心分數）。
   - cowork/你：只做「這筆是不是 ACG、要不要收」的判斷，通過才進正式資料。
   - 這樣機械活（抓取、去重、缺口偵測）全自動，cowork 只做人類擅長的判斷，且每筆判斷有紀錄可稽核，解掉 2.3 最後一列的「判斷不一致」。
4. **分類升級**：關鍵字表改成「關鍵字命中就自動 ACG；沒命中但來源可疑 → 進候選待審」，而不是直接歸「其他文化」漏掉。

## 三、其他你可能沒注意到、但值得改的地方

### 3.1 部署與可分享（成長引擎，目前是零）
- **沒有正式部署證實**：先把 `public/` 上 **GitHub Pages / Cloudflare Pages**，取得穩定網址。純靜態站，成本近乎零，且能配合 CI 自動發佈。
- **沒有任何 SEO / 社群分享 meta**：`<head>` 裡**沒有** `meta description`、`og:title`、`og:image`、`twitter:card`。結果：貼到 Line/FB/Threads 沒有預覽圖與描述，Google 搜「台灣 ACG 展覽」也搜不到你。對一個靠口碑擴散的工具，這等於**關掉了成長管道**。加幾行 meta 幾乎零成本，投報比極高。
- **可考慮 PWA**：加 `manifest.json` + 簡單 service worker，讓使用者「加到主畫面」像 App，且離線可看上次資料。ACG 追展族手機使用比例高。

### 3.2 效能
- **單檔 730KB、且資料被載入兩次**：`venues.json`（約 88KB）同時**內嵌**在 HTML 的 `let DATA = ...`（約 88KB）**又** `fetch('venues.json')` 再抓一次。線上版等於下載兩份同樣資料。內嵌備援是為了 `file://` 離線用，但線上使用者不需要——可考慮線上版不內嵌、或內嵌一個精簡版。
- **地圖函式庫走 unpkg CDN**：Leaflet/MapLibre 從 `unpkg.com` 載入。unpkg 掛了或被牆，**整張地圖開不出來**。建議把函式庫在地化（打包進 `public/`）或至少加 fallback。

### 3.3 行動裝置 / 無障礙（a11y）
- **封鎖了雙指縮放**：`<meta viewport ... maximum-scale=1.0>`。這會**禁止使用者在手機上放大**，對視力不佳者是無障礙缺陷，Lighthouse a11y 會直接扣分。地圖本身有縮放，但頁面文字/卡片放不大。建議拿掉 `maximum-scale=1.0`。
- 圖釘/按鈕的觸控目標、對比度、`aria-*` 可再過一輪 Lighthouse。

### 3.4 資料品質與可信度
- **KV 圖床過期**（同 2.3）：建議把 KV 圖**下載到本地** `public/`（像 logo 一樣做縮圖快取），不要直連 IG CDN，避免整片破圖。
- **151 場活動全無簡介（`h` 空）**：資訊密度不足以支撐「要不要去」的決策，見 1.2。
- **30 場無結束日**：`即將結束` 篩選、提醒功能都依賴結束日，缺了就失效。

### 3.5 工程健康度
- **沒有任何自動化測試 / CI 檢查**：`venues.json` 的 schema、前端能不能載入、`update_all.py` 能不能 import，目前都靠人工。至少加一個 GitHub Actions：`py_compile` 全後台 + `json.load(venues.json)` + 檢查必要欄位存在。這也順便逼出部署管線。
- **前端是 730KB 單檔手改**：`DECISIONS.md` 記載已刻意放棄 build 流程、接受手改。理解這個取捨，但檔案越大越容易改壞（且無測試接不住）。長期若功能變多，值得重新評估「極簡 build（只做資料注入與壓縮）」。
- **隱私**：目前純前端無追蹤，這反而是優點；若之後加 analytics，建議用隱私友善方案（如 Plausible/Cloudflare Web Analytics）並在頁面標註。

## 四、優先級路線圖

**P0（沒有它就沒有產品，先做這三件）**
1. 正式部署 `public/` 到 GitHub Pages / Cloudflare Pages，取得公開網址。
