## 後台模組對應

- 內容管理：`data/manual/`
- 資料管線：`backend/update_all.py`、`backend/refresh_venues.py`、`backend/collect_*.py`
- 圖像資產：`public/logos/` 與 `data/logos/`
- 地圖發布：`public/`
- 維護報表：`backend/report_status.py` 與 `data/reports/`
- 排程與工作狀態：`ops/` 與 `runtime/`

## 活動主視覺（KV）自存流程

> 2026/07/20 起改為「完整自存模式」。給其他 agent 的重點：`venues.json` 的 `e.img` 現在**大多是站內路徑 `kv/<hash>.<副檔名>`，不是可外連的 http 網址**。

- **`e.img` 的取值**：能成功下載的遠端主視覺，一律自存到 `public/kv/` 並把 `e.img` 改寫成站內相對路徑 `kv/<16位hash>.<ext>`；下載失敗才保留原遠端網址（容錯，不寫壞資料）。前端 `safeUrl` 會正確解析 `kv/...`，**前端不用改**。
- **檔名規則**：`sha1(stable_event_key(場館, 標題, 起始日))[:16] + 副檔名`，跨執行穩定，同一活動永遠對到同一檔（冪等：本地檔已存在則只改寫路徑、不重抓）。
- **產生時機**：`update_all.py` 最後一輪 `refresh_venues.py` 之後、`sync_embed` 之前跑 `download_event_kv.py --all`。更早的管線步驟看到的 `e.img` 仍是遠端 URL，最後這步才本地化。
- **清理**：活動過期被 `refresh_venues.py` 移除後，其 KV 不再被引用，`download_event_kv.py` 會把 `public/kv/` 內未被任何活動引用的孤兒圖自動刪除。
- **模式旗標**：`download_event_kv.py` 預設只自存會過期的 IG/FB CDN 圖；`--all` 才自存所有遠端圖（管線走 `--all`）。相關快取 `data/cache/event_kv_cache.json` 只存「連結→圖片網址」的抽取結果，不存圖檔本身。
- **注意**：`public/kv/` 是自動產物，別手動增刪（過期會被 GC）；此模式會讓資料夾隨每日 CI 成長到約 150+ 張圖並一起 commit，repo 會變大，這是「不依賴外站、避免破圖」的取捨。

## 路徑規則

- 後台程式不可再寫死根目錄檔名，請使用 `backend/paths.py` 的 `P("檔名")`。
- `venues.json` 裡的 logo 值維持 `logos/...`，因為這是前端公開路徑；實體檔案位於 `public/logos/...`。
- `venues.json` 裡的 `e.img` 大多是站內路徑 `kv/...`（同 logo，實體檔在 `public/kv/...`）；別假設它一定是可外連的 http 網址。
- `data/manual/` 視為人工資料與後台管理表單的未來來源；自動管線不應無故覆蓋人工修正。
- `data/generated/`、`data/cache/`、`runtime/` 可由程式重建或更新。
- 瀏覽器 profile 內部子資料夾由 Chrome/Playwright 自動管理，不逐層加入專案說明檔；說明集中放在 `runtime/profiles/` 與其第一層 profile 目錄。

## 常用指令

```bash
# 更新資料
python3 backend/update_all.py

# 本機預覽
cd public
python3 -m http.server 8000
# 開 http://localhost:8000/taiwan-exhibition-map.html

# 檢查核心程式與資料
python3 -m py_compile backend/update_all.py backend/refresh_venues.py backend/collect_logos.py
python3 -c 'import json; json.load(open("public/venues.json", encoding="utf-8")); print("venues json ok")'
```
